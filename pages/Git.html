Работа с Git

Начальный уровень 25 минут 6 разделов

Git — это распределенная система контроля версий, которая позволяет отслеживать изменения в исходном коде, работать в команде и управлять историей разработки. Основа любого современного проекта!

Что такое Git?

Git — это система контроля версий (VCS), созданная Линусом Торвальдсом для разработки ядра Linux. Она помогает:

Хранить историю

Все изменения кода за всё время

Работать в команде

Без конфликтов и потерь данных

Создавать ветки

Параллельная разработка функций

Откатывать изменения

Возврат к любой точке истории

Интересный факт

Git был создан в 2005 году после того, как компания BitKeeper отозвала бесплатную лицензию для разработчиков ядра Linux. Линус Торвальдс написал Git всего за 2 недели!

Локальный репозиторий

Все данные хранятся на вашем компьютере:

  • Полная история проекта
  • Работа без интернета
  • Быстрые операции
  • Сохранение всех версий файлов

Удаленный репозиторий

Сервер для совместной работы:

  • GitHub — самый популярный хостинг
  • GitLab — с CI/CD инструментами
  • Bitbucket — от Atlassian
  • Обмен изменениями между разработчиками

Рабочий процесс

Типовая схема работы:

Рабочая директория → Индекс → Репозиторий

1. Работаете с файлами
2. Добавляете изменения в индекс (stage)
3. Фиксируете изменения (commit)
4. Отправляете на сервер (push)

Установка и настройка Git

Перед началом работы нужно установить и настроить Git на вашем компьютере.

Проверка установки:

# Проверить, установлен ли Git
git --version

# Если установлен, вы увидите что-то вроде:
# git version 2.40.1

Скачать Git:

  • Windows: git-scm.com/download/win
  • macOS: brew install git или скачать с сайта
  • Linux: sudo apt install git (Ubuntu/Debian)

Настройка Git:

После установки необходимо настроить основные параметры:

# Настроить имя пользователя (обязательно!)
git config --global user.name "Ваше Имя"

# Настроить email (обязательно!)
git config --global user.email "ваш.email@example.com"

# Включить цветной вывод
git config --global color.ui auto

# Настроить редактор по умолчанию
git config --global core.editor "code --wait"  # VS Code
# или
git config --global core.editor "nano"         # Nano
# или
git config --global core.editor "vim"          # Vim

Проверка настроек:

Чтобы увидеть все текущие настройки, выполните:

git config --list

Эта команда покажет все установленные конфигурации.

Первые команды Git

Начнем с основных команд, которые понадобятся каждый день.

1

Инициализация репозитория

Создать новый репозиторий или начать отслеживание существующего проекта:

# Создать новый репозиторий в текущей папке
git init

# Репозиторий будет создан в скрытой папке .git
# Проверить статус репозитория
git status
2

Добавление файлов

Подготовить файлы к коммиту (добавить в staging area):

# Добавить все измененные файлы
git add .

# Добавить конкретный файл
git add index.html

# Добавить все файлы с расширением .js
git add *.js

# Добавить все файлы в папке src
git add src/
3

Создание коммита

Зафиксировать изменения с комментарием:

# Создать коммит с сообщением
git commit -m "Добавлена главная страница"

# Добавить и закоммитить одним действием
git commit -am "Обновлены стили"

# Просмотреть историю коммитов
git log --oneline

# Подробный просмотр истории
git log --graph --oneline --all
Сообщения коммитов должны быть понятными и описывать, что было сделано!

Практикуем основные команды:

# Создадим проект и поработаем с Git

# 1. Создаем папку проекта
mkdir my-project
cd my-project

# 2. Инициализируем репозиторий
git init

# 3. Создаем файлы
echo "# Мой проект" > README.md
echo "console.log('Hello Git!')" > app.js

# 4. Проверяем статус
git status

# 5. Добавляем файлы
git add .

# 6. Создаем коммит
git commit -m "Initial commit: добавлены README и app.js"

# 7. Смотрим историю
git log --oneline

Ветвление (Branching)

Ветки позволяют работать над разными задачами параллельно, не мешая основной разработке.

Основная ветка

Главная ветка проекта (обычно main или master)

# Посмотреть все ветки
git branch

# Создать новую ветку
git branch feature-login

# Переключиться на ветку
git checkout feature-login
# или
git switch feature-login

Слияние веток

Объединение изменений из разных веток

# Перейти в основную ветку
git checkout main

# Влить изменения из feature ветки
git merge feature-login

# Если есть конфликты, Git сообщит
# Нужно их разрешить и сделать коммит

Удаление веток

Очистка после завершения работы

# Удалить локальную ветку
git branch -d feature-login

# Принудительное удаление
git branch -D feature-login

# Удалить удаленную ветку
git push origin --delete feature-login

Конфликты слияния

Решаем конфликты при merge

# Если при слиянии возник конфликт
<<<<<<< HEAD
ваша версия кода
=======
версия кода из ветки
>>>>>>> feature-login

# Редактируем файл, оставляя нужный код
# Затем добавляем и коммитим
git add файл_с_конфликтом
git commit -m "Разрешен конфликт слияния"

Рабочий процесс с ветками:

Типовая схема работы с ветками в проекте:

  • main — стабильная версия, готовый продукт
  • develop — ветка разработки
  • feature/* — ветки для новых функций
  • hotfix/* — срочные исправления для main
  • release/* — подготовка к выпуску

Работа с удаленным репозиторием

GitHub, GitLab и другие хостинги позволяют хранить код в облаке и работать в команде.

Команда Описание Пример Когда использовать
git remote add Добавить удаленный репозиторий git remote add origin URL Первый раз привязываем локальный репозиторий к GitHub
git push Отправить изменения на сервер git push origin main После коммитов, чтобы поделиться изменениями
git pull Забрать изменения с сервера git pull origin main Перед началом работы, чтобы получить свежие изменения
git clone Скопировать репозиторий git clone URL Начать работу с существующим проектом
git fetch Получить изменения (без слияния) git fetch origin Посмотреть, что изменилось на сервере
git remote -v Показать удаленные репозитории git remote -v Проверить, куда отправляются изменения

Работа с GitHub:

# Пример полного цикла работы с GitHub

# 1. Клонируем существующий репозиторий
git clone https://github.com/username/project.git
cd project

# 2. Создаем новую ветку для задачи
git checkout -b add-feature

# 3. Работаем над кодом, делаем изменения
echo "Новая функция" > new-feature.js

# 4. Добавляем и коммитим изменения
git add new-feature.js
git commit -m "Добавлена новая функция"

# 5. Отправляем ветку на GitHub
git push origin add-feature

# 6. На GitHub создаем Pull Request
# 7. После одобрения сливаем ветку
git checkout main
git pull origin main
git merge add-feature
git push origin main

# 8. Удаляем отработанную ветку
git branch -d add-feature
git push origin --delete add-feature

Типовые рабочие процессы

Рассмотрим популярные подходы к организации работы с Git.

Git Flow

Строгая модель для больших проектов:

  • main — стабильные релизы
  • develop — основная разработка
  • feature/* — новые функции
  • release/* — подготовка релиза
  • hotfix/* — срочные исправления
# Начало работы над новой функцией
git checkout develop
git checkout -b feature/new-auth

# Завершение функции
git checkout develop
git merge --no-ff feature/new-auth
git branch -d feature/new-auth

GitHub Flow

Упрощенный подход для непрерывного развертывания:

  • Только ветка main как основа
  • Ветки для всех изменений
  • Pull Request для code review
  • Развертывание после мержа
# Простой рабочий процесс
git checkout main
git pull origin main
git checkout -b fix-bug-123

# После работы
git push origin fix-bug-123
# Создаем Pull Request на GitHub
# После ревью и тестов - merge

Trunk Based Development

Современный подход для частых релизов:

  • Все работают в main/trunk
  • Короткоживущие ветки (1-2 дня)
  • Частые мелкие коммиты
  • Функциональные флаги вместо веток
# Короткий цикл разработки
git checkout main
git pull origin main

# Небольшие изменения, сразу в main
# или короткая ветка
git checkout -b small-change
# Быстрая работа и мерж

Какой подход выбрать?

  • Git Flow — для крупных проектов с релизами
  • GitHub Flow — для веб-приложений и сервисов
  • Trunk Based — для стартапов и DevOps

Практическое задание

Создадим свой первый проект на GitHub от начала до конца:

Задача: Первый репозиторий на GitHub

Начальный уровень

Выполните все шаги для создания и настройки репозитория:

  1. Создайте аккаунт на GitHub (если нет)
  2. Создайте новый репозиторий на GitHub
  3. Склонируйте его на локальный компьютер
  4. Добавьте файлы проекта
  5. Создайте ветку для новой функции
  6. Сделайте коммит и отправьте изменения
  7. Создайте Pull Request и слейте изменения

Подсказки:

  • Используйте git clone для копирования репозитория
  • git checkout -b создаст и переключит на новую ветку
  • git push -u origin branch-name отправит ветку на GitHub
  • На GitHub в интерфейсе создавайте Pull Request

Пошаговые команды:

Результат выполнения команд появится здесь

Полный сценарий работы:

# === ПОЛНОЕ РЕШЕНИЕ ===

# 1. НАСТРОЙКА GIT (делается один раз)
git config --global user.name "Ваше Имя"
git config --global user.email "ваш.email@example.com"

# 2. СОЗДАНИЕ ЛОКАЛЬНОГО РЕПОЗИТОРИЯ
mkdir my-first-project
cd my-first-project
git init

# 3. СОЗДАНИЕ ФАЙЛОВ ПРОЕКТА
echo "# Мой первый проект на Git" > README.md
cat > index.html << 'EOF'



    Мой проект
    


    
Привет, Git!

    


EOF

echo "console.log('Проект запущен!')" > app.js
echo "body { font-family: Arial; margin: 20px; }" > style.css

# 4. ПЕРВЫЙ КОММИТ
git add .
git commit -m "Initial commit: базовая структура проекта"

# 5. ПОДКЛЮЧЕНИЕ К GITHUB
# Предварительно создайте репозиторий на GitHub
git remote add origin https://github.com/username/my-first-project.git
git branch -M main
git push -u origin main

# 6. РАБОТА НАД НОВОЙ ФУНКЦИЕЙ
git checkout -b feature/add-header
cat >> index.html << 'EOF'


    

        Главная
        О нас
    



EOF

git add index.html
git commit -m "Add navigation header"
git push origin feature/add-header

# 7. СОЗДАНИЕ PULL REQUEST
# Перейдите на GitHub в вашем репозитории
# Нажмите "Compare & pull request" для ветки feature/add-header
# Добавьте описание и создайте PR
# После ревью нажмите "Merge pull request"

# 8. ОБНОВЛЕНИЕ ЛОКАЛЬНОЙ КОПИИ
git checkout main
git pull origin main

# 9. УДАЛЕНИЕ ОТРАБОТАННОЙ ВЕТКИ
git branch -d feature/add-header
git push origin --delete feature/add-header  # опционально

# 10. ПРОВЕРКА РЕЗУЛЬТАТА
git log --oneline --graph --all
Советы по улучшению:
  • Используйте .gitignore для исключения ненужных файлов
  • Пишите подробные сообщения коммитов
  • Делайте частые мелкие коммиты вместо одного большого
  • Всегда делайте git pull перед началом работы
